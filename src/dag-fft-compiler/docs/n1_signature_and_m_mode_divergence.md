# The n1 signature zoo and the M-mode path divergence

This note records a structural problem surfaced while trying to benchmark
R32/R64/R128 in the three M modes (M-off / M-fence / M-on). The short
version: there is no single "n1 codelet" — there are at least three
incompatible call signatures, and which one you get depends on the
generation path, and the M-mode knobs are not uniform across those
paths. One path can't even emit a pin. This is why a standalone driver
that works for one radix segfaults on another, and why "what M mode is
R128" had no clean answer.

## 1. Three n1 layouts in play

There are at least three distinct n1 (no-twiddle) leaf signatures
floating around, and the registry/executor pairing decides which one a
given consumer expects:

1. **6-arg rio in-place**
   `(rio_re, rio_im, tw_re, tw_im, ios, me)`
   Single read-in-out buffer used as its own pass-1/pass-2 scratch.
   This is what `gen_radix … --in-place` (without `--oop`) emits.

2. **7-arg OOP (in==out convention)**
   `(in_re, in_im, out_re, out_im, is, os, me)`
   The form the generic executor's call site uses; it runs in-place by
   passing `in == out` and `is == os`
   (see `core/executor_generic.h` ~line 66 and ~207).

3. **11-arg OOP-buffer**
   `(in_re, in_im, out_re, out_im, tw_re, tw_im,
     in_leg_stride, in_group_stride, out_leg_stride, out_group_stride, me)`
   The Bailey v3_t1 OOP-buffer codelet, e.g.
   `radix128_n1_oop_fwd_avx512_UG_UG` in
   `codelets/oop/avx512/radix128_n1_oop_avx512.c`.

These are not interchangeable. A standalone driver written against one
signature will pass the wrong arguments to another (wrong pointer in a
size slot, missing stride args), which is exactly the class of bug that
produced the R128 segfault: the driver was built for the 7-arg OOP
convention, but the `--in-place` R128 it was handed was the 6-arg rio
form, so the call ran off into garbage addresses.

**Takeaway:** "the n1 codelet" is ambiguous. Always pin down which of
the three signatures a codelet was generated in, and match the driver
(or executor) to it, before trusting any run.

## 2. The decisive finding: pinning is not wired on the OOP path

The provenance header on the generated OOP R128 codelet states it
plainly:

```
Regalloc+pinning: not wired on the OOP path (render-convention blocker, section 36)
Value fences: true (Pipeline-computed prep.fence_enabled)
```

So on the OOP path, **pinning literally cannot be emitted** — the M-on
(pin) mode does not exist there. The only M knob that path has is the
fence, driven by the two-rule `prep.fence_enabled`
(`docs/fence_pin_decomposition.md`).

This means the set of available M modes depends on the path:

| Path | Generator selector | M-off | M-fence (fence only) | M-on (pin + fence) |
|---|---|:---:|:---:|:---:|
| OOP (`codelet_oop.ml`) | `--oop` | yes | yes | **no — pin not wired** |
| In-place (`emit_c.ml`) | `--in-place` (shipped set) | yes | yes | yes |

Consequences:

- **OOP path:** the only meaningful comparison is M-fence vs M-off,
  because pin is *structurally absent*, not merely policy-disabled.
  Asking "does the M rule apply on OOP" reduces to "does the fence
  help." There is no third data point to collect.

- **In-place path (`emit_c.ml`):** all three modes exist. The gate
  (`regalloc_enabled` at `emit_c.ml` ~line 1020) plus `VFFT_PIN_FORCE`
  can produce a pinned variant. This is the path the shipped
  `codelets/inplace/` set uses (generated by `generate_codelets.sh`
  with `--in-place --su`, not `--oop`), and therefore the path the
  MKL head-to-head ran on.

## 3. Why this matters for the M experiment

The whole point of the three-way M-off / M-fence / M-on benchmark is to
check whether the fence-vs-pin decomposition
(`docs/fence_pin_decomposition.md`, measured on a per-radix variant
basis) also holds for the in-place codelets that actually ship.

That experiment is **only meaningful on the in-place path**, because
that is the only path where all three modes can be generated. On the
OOP path, M-on can't be produced, so a "three-way" there would silently
collapse to a two-way and any "M-on" column would be a mislabeled fence
variant.

So:

- **R32, R64 in-place:** full three-way is generatable and the
  standalone harness drives them cleanly. These are the codelets that
  ship and were benched against MKL, so they are the codelets the doc's
  claim should be checked against.

- **R128:** the in-place (6-arg rio) form is the segfault case until a
  matching driver is written; the OOP form is 11-arg and has no pin to
  test anyway. R128 is experimental and not in the registry, so it adds
  no shipped-codelet evidence beyond confirming the R32/R64 trend.

## 4. The deeper issue (separate from the benchmark)

The M decision is embedded in **two** places that have **drifted**:

- `codelet_oop.ml` (OOP path): clean two-rule `fence_enabled`, pin
  deferred/unwired. Matches the policy doc.
- `emit_c.ml` (in-place path): older gate that still carries the
  `is_avx2 && is_n1 && radix >= 16` pin clause and still honors
  `VFFT_PIN_FORCE`.

Because the two emitters compute M differently, the same logical codelet
can come out with different M behavior depending on which path generated
it, and an env override (`VFFT_PIN_FORCE`) can push the in-place path
off-policy. That divergence — not any single codelet — is the
"what if someone forgets" risk: M is embedded, but inconsistently, and
the OOP path's "pin not wired" status is an additional asymmetry on top.

Any unification work should (a) make the in-place `emit_c.ml` gate
compute the same two rules as `codelet_oop.ml`, and (b) decide whether
the OOP path's missing pin is intended (the policy says pin only helps
`log3 ∧ avx512 ∧ R≤32`, which the OOP path may simply never hit) or a
gap to close. Until then, every M claim must name the path it was
measured on.
