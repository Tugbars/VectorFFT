Architecture Overview

VectorFFT decomposes any DFT size N into a chain of small fixed-radix stages, each handled by a specialized SIMD codelet. The execution layer coordinates factorization, twiddle application, stage ordering, and output permutation.

Execution Strategies

The planner selects an execution strategy based on the number of stages in the factorization. This mirrors Intel MKL's empirical approach.

N factored → S stages

  if S = 1 → DIRECT: single codelet call, no twiddles
  if S = 2..3 → DIT: row-column + twiddle + digit-reversal permutation
  if S ≥ 4 → STOCKHAM: autosort, no final permutation needed
Why this threshold? empirical

Stockham autosort writes output in natural order (no permutation pass) but adds a gather/scatter transpose per stage. For 2-3 stages, the DIT approach avoids this overhead — the digit-reversal permutation at the end costs O(N) but only once. At 4+ stages, Stockham's automatic reordering wins because the per-stage transpose cost is amortized over more computation.

Row-Column DIT (2-3 stages)

Process stages from outermost to innermost. Each stage applies a codelet at the current stride, then multiplies by precomputed twiddle factors. The final digit-reversal permutation reorders the output.

for s = S-1 downto 0:
    R = radix[s], K = stride[s]
    for each group g:
        codelet_R(data + g*R*K, output + g*R*K, K)
        if K > 1:
            apply_twiddles(output + g*R*K, tw_table[s], R, K)  // SIMD dispatch

digit_reversal_permute(output, perm_table, N)

Stockham Autosort (4+ stages)

Each stage reads from source buffer, writes to destination buffer with transposed index mapping. Twiddle = Waccumulatedk·inner where accumulated = product of factors through current stage. Output is in natural order — no permutation needed.

Critical twiddle formula bug that took 3 iterations

The twiddle is not WNk·inner. It is Waccumulatedk·inner where accumulated = R₀ × R₁ × ... × Rs. For 2-stage FFTs accumulated = N, so the "standard" formula works by accident. For 3+ stages it is essential to use the partial product. This was the hardest bug in the entire planner.

Prime Radix Codelets (genfft)

For primes, the Cooley-Tukey factorization doesn't apply. FFTW's genfft OCaml code generator produces expression-optimized straight-line code that minimizes total operation count through algebraic simplification and common subexpression elimination.

Macro-body architecture key innovation

One butterfly body, three ISA instantiations. The arithmetic is written once using abstract macros (ADD, SUB, MUL, FMA, LD, ST), then instantiated for scalar, AVX2, and AVX-512 by redefining the macros as C operators or intrinsics. This eliminates the 3× code duplication of the radix-11 approach.

// Define body ONCE:
#define R17_BUTTERFLY_BODY do {     \
    V T1 = ADD(LD(&ri[1*K+k]), LD(&ri[16*K+k]));  \
    V T2 = SUB(LD(&ri[1*K+k]), LD(&ri[16*K+k]));  \
    ...                                          \
    ST(&ro[0*K+k], ADD(T_dc, T_sum));             \
} while(0)

// Scalar instantiation:
#define V      double
#define LD(p)  (*(p))
#define ADD(a,b) ((a)+(b))
for(k=0; k<K; k+=1) { R17_BUTTERFLY_BODY; }

// AVX-512 instantiation:
#define V      __m512d
#define LD(p)  _mm512_load_pd(p)
#define ADD    _mm512_add_pd
for(k=0; k<K; k+=8) { R17_BUTTERFLY_BODY; }

Packed layout + twiddle walking

For large K, data is repacked from K-strided to contiguous blocks of SIMD width T (4 for AVX2, 8 for AVX-512). This ensures every load/store is aligned and sequential. Twiddle walking pre-applies external twiddle factors block-by-block, avoiding redundant memory passes.

Why genfft beats hand-built for primes validated

For prime N, there's no natural Cooley-Tukey decomposition. FFTW's genfft performs global algebraic optimization across the entire DFT-N, finding common subexpressions that a butterfly-stage approach cannot. The 160-op count for DFT-11 vs. a naive 11² = 121 complex multiplies demonstrates the power of this approach. For N=17,19,23 — FFTW doesn't even ship hard-coded codelets, falling back to a generic O(N log N) Rader solver at runtime. Our hard-coded genfft codelets skip that overhead entirely.

Composite Radix Codelets (Cooley-Tukey)

For power-of-2 and small composite sizes (2–128), hand-built split-radix codelets with Python-generated butterfly patterns outperform genfft's flattened DAG approach.

genfft loses at composites experimentally confirmed

Tested DFT-32: genfft produces a 629-line straight-line block with 100 stack variables and 424 ops. Our split-radix codelet decomposes into compact butterfly stages (~50 lines each), reuses twiddle factors across stages, and lets the compiler schedule each stage independently. Result: genfft DFT-32 AVX2 = 0.86× FFTW at K=1024, while our split-radix DFT-32 AVX2 = 0.96× FFTW.

// Strategy for VectorFFT:
//   Primes (11, 13, 17, 19, 23):  genfft DAG codelets
//   Composites (2–128):           hand-built split-radix codelets
//
// The hierarchical decomposition of Cooley-Tukey naturally fits
// the hardware's register file and µop cache for composites.
// genfft's global optimization is the right tool for primes
// where no such decomposition exists.

Bluestein Fallback (Primes > 23)

For prime factors larger than 23 (where no hard-coded codelet exists), Bluestein's chirp-z algorithm converts the DFT into a circular convolution via power-of-2 FFTs.

Batch execution (Option C — interleaved)

For K independent DFTs: all K chirp multiplies execute together (SIMD-parallel over K), all K zero-padded sequences form one K-strided size-M array, the internal FFT sees K batched DFTs. This maximizes SIMD utilization at every step.

// Pluggable internal FFT via function pointer:
typedef void (*vfft_fft_fn)(double *re, double *im,
                            size_t N, size_t K, int direction);

vfft_bluestein_plan *plan = vfft_bluestein_create(37, my_optimized_fft);
// Internal FFTs of size 64/128/256 use your AVX-512 codelets
// Chirp/pointwise have their own AVX-512 + AVX2 paths

Twiddle Factor Management

Precomputed tables

Twiddle factors are computed once at plan creation, stored as contiguous 64-byte-aligned arrays per stage. Total memory is O(N). The execution hot loop does zero trigonometric function calls.

40× speedup from precomputation measured
R=17 K=128 (2032 twiddles):
  precomputed + AVX-512:   629 ns
  on-the-fly cos/sin:    25,321 ns
  speedup:                 40.3×

SIMD twiddle application

The inner dimension K is the SIMD-vectorizable axis. When K is aligned to SIMD width, the twiddle multiply uses FMA instructions — 4 instructions per 8 values (AVX-512) or 4 values (AVX2). Unaligned K falls back to scalar with no performance cliff.

// Dispatch: K%8==0 → AVX-512, K%4==0 → AVX2, else → scalar
vfft_apply_twiddles_dispatch(data_re, data_im, tw_re, tw_im, R, K);

// AVX-512 inner loop:
__m512d a   = _mm512_load_pd(&xr[k]);
__m512d b   = _mm512_load_pd(&xi[k]);
__m512d w_r = _mm512_load_pd(&wr[k]);
__m512d w_i = _mm512_load_pd(&wi[k]);
_mm512_store_pd(&xr[k], _mm512_fmsub_pd(a, w_r, _mm512_mul_pd(b, w_i)));
_mm512_store_pd(&xi[k], _mm512_fmadd_pd(a, w_i, _mm512_mul_pd(b, w_r)));

Planner Design

Factorization

Greedy largest-first from the supported radix set, preferring power-of-2 factors. This minimizes stage count (each stage has overhead) and keeps inner K large for SIMD alignment.

// Supported radixes in preference order:
128, 64, 32, 16, 8, 4, 2,     // powers of 2 (best codelets)
9, 10, 6,                       // small composites
23, 19, 17, 13, 11, 7, 5, 3,   // primes (large to small)

// Examples:
N = 6528 = 2⁷ × 3 × 17  → [3, 17, 128]   // 3 stages
N = 2310 = 2×3×5×7×11    → [2, 3, 5, 7, 11] // 5 stages → Stockham
N = 1024 = 2¹⁰           → [8, 128]        // 2 stages
N = 37 (prime > 23)       → Bluestein       // M=128 internal FFT

Codelet registry

Function-pointer table indexed by radix. Naive O(N²) fallbacks are registered at init, then overridden with optimized codelets. Each dispatch wrapper checks K alignment and selects AVX-512 → AVX2 → scalar.

vfft_codelet_registry reg;
vfft_register_all(®);  // auto-detects ISA, registers best codelets

// Drop in your hand-built split-radix:
vfft_registry_set(®, 8,  your_radix8_fwd,  your_radix8_bwd);
vfft_registry_set(®, 64, your_radix64_fwd, your_radix64_bwd);

vfft_plan *plan = vfft_plan_create(N, ®);
vfft_execute_fwd(plan, in_re, in_im, out_re, out_im);
