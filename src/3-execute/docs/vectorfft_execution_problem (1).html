01 What Happens Now

VectorFFT uses Stockham autosort: each stage processes the entire array, reading from one buffer and writing to another. Stages are independent — stage 1 doesn't know stage 0 exists.

for each stage s = 0 to S-1:
  for each group g = 0 to n_groups-1:
    codelet(src + g*R*K, dst + g*R*K, twiddles, K)
  swap(src, dst)

// Every stage scans ALL N elements.
// Each stage evicts the previous stage's cached data.

The Cascade of Cache Misses

For N=4096 (64KB of complex data), your L1 cache is 48KB. Every stage reads 64KB and writes 64KB — the entire array doesn't fit. With 4 stages, that's 8 full scans of an array that exceeds L1.

02 The Stride Problem

Inner stages (small K) access contiguous data — great cache behavior. Outer stages (large K) are the killer. Each codelet reads R values spaced K elements apart.

Interactive: How stride grows per stage

Stage 0: R=32, K=1. The codelet reads 32 contiguous elements. All fit in 1-2 cache lines. Perfect locality.

At K=512, a radix-4 codelet reads 4 values spanning 16KB — one third of L1. Each group pollutes the cache with 4 widely-separated cache lines, and successive groups share none of them.

03 The Permutation Tax

Before any computation starts, DIT requires a digit-reversal permutation — a random-access scatter over the entire array. For N=32768, that's 32768 random reads across 512KB. Every single access is an L1 miss.

Current: DIT Forward

Input: Random scatter (cache-hostile)

buf[i] = input[perm[i]]

Every read misses L1 — the permutation is pseudo-random. This single pass costs 2000-3000ns at large N.

Output: Natural order ✓

FFTW: DIF Forward + DIT Backward

Input: Sequential read (prefetcher-friendly)

Output: Bit-reversed (doesn't matter — backward expects it)

The permutation never materializes as an explicit pass. DIF output feeds directly into DIT input. Zero cost.

05 The Three Symptoms

Symptom 1

Zero data reuse between stages

Each stage reads the full array, processes it, writes the full array. The next stage reads the same array — but by now it's been evicted from L1 by the write pass. Every stage starts with a cold cache.

Symptom 2

Strided access at large K

Outer stages have K = 128, 256, 512+. The codelet reads R values at stride K — each value from a different cache line, spanning R×K×8 bytes. At R=32, K=512, that's 128KB per group — 2.7× your entire L1.

Symptom 3

Permutation as separate random-access pass

The DIT input permutation scatters N values randomly across the buffer. For N=32768, that's 32768 random reads — every one an L1 miss. This costs 20-30% of the total forward execution time.

06 The Solution

Four changes. The codelets stay. The butterfly math stays. Only the execution layer changes.

Step 1 — Unify Generators

One generator per radix family, parameterized execution mode

Currently 15+ separate Python generators. Unify into 3 families: power-of-2 (R=2,4,8,16,32,64), odd composites (R=3,5,7,10,20,25), primes (R=11,13,17,19,23). Each generator emits multiple execution modes from the same butterfly math.

# One generator, three execution modes:
def emit_codelet(R, variant, execution_mode):
  if mode == "strided":   # current: in[n*K+k]
  if mode == "contiguous": # new: data pre-gathered to scratch
  if mode == "buffered": # new: gather → butterfly → scatter

  emit_butterfly(R, dag_or_ct) # ← same math, always
Step 2 — Buffered Execution

Gather strided data into L1-sized scratch, run codelet, scatter back

For outer stages with large K, copy the R strided values into a small contiguous buffer, run the butterfly on L1-hot contiguous data, then scatter the results back. The copy is sequential and predictable — the prefetcher handles it perfectly.

Step 3 — DIF/DIT Fusion

Eliminate the permutation pass entirely

Use DIF for the outer decomposition and DIT for the inner sub-problems. DIF produces bit-reversed output. DIT expects bit-reversed input. They cancel at each boundary — no explicit permutation ever happens.

// DIF outer level:
  Reads natural-order input
  Produces bit-reversed output within each block

// DIT inner level:
  Expects bit-reversed input ← exactly what DIF just produced!
  Produces natural-order output

// Permutation cost: ZERO

This is why having both DIT and DIF codelets was the right investment. They're the prerequisite for this optimization. Now the execution layer needs to exploit them.

Step 4 — Sub-Problem Completion

Process each L1-sized sub-DFT through all its stages before moving on

Instead of running all sub-problems through stage 0, then all through stage 1, etc. — run one sub-problem through all stages while it's still in L1. Then move to the next. This is the FFTW approach adapted to our codelet infrastructure.

// Before: stage-by-stage (current)
for stage in [0, 1, 2, 3]:
  for group in all_groups: // ← scans entire array per stage
    codelet(group)

// After: sub-problem completion (new)
for sub_problem in all_sub_problems:
  for stage in [0, 1, 2]: // ← data stays in L1 across stages
    codelet(sub_problem, stage)
for stage in [3, 4]: // outer stages, full-N (unavoidable)
  codelet(all_data, stage)
