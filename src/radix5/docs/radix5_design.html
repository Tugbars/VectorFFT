VectorFFT Technical Report

Radix-5 Codelet
Architecture & Design

Eliminating the serial twiddle derivation chain — from 15-cycle latency to zero, without sacrificing cache behavior at high K.

March 2026 Target: AVX2 / AVX-512 Status: Production

01 — The Problem

Radix-5 was VectorFFT's weakest codelet. Sizes with heavy factor-of-5 content (N=5000, 10000, 20000, 40000) consistently lost to FFTW by 10–25%. The root cause was a serial twiddle derivation chain that stalled the FMA pipeline every iteration.

The log3 chain

The original R=5 codelet loaded a single twiddle base W¹ and derived the remaining three through chained complex multiplications:

// Old approach: log3 derivation (serial dependency chain)
W¹ ← load(tw_re[0], tw_im[0])   // 1 load, ~4 cycles
W² = W¹ × W¹                      // depends on W¹, +5 cycles
W³ = W¹ × W²                      // depends on W², +5 cycles
W⁴ = W¹ × W³                      // depends on W³, +5 cycles
//                                    Total: 1 load, 15 cycle latency

Each complex multiplication (2 FMA + 1 MUL) has ~5 cycles of latency on modern Intel cores. With a 3-deep serial chain, the butterfly couldn't start until 15 cycles after the twiddle load — every single k-iteration.

For radixes like R=32 (31 twiddle rows), log3 makes sense: deriving 29 twiddles from 2 bases saves 29 cache-line loads. But R=5 has only 4 twiddle rows. The tradeoff:

A bad trade. Those 3 saved loads cost ~4 cycles each (L1 hit). Net cost: +3 cycles per iteration, compounding across thousands of k-steps.

02 — The Solution: Flat Twiddle Loads

Load all 4 twiddles independently. Zero derivation latency. The prefetcher handles the strided access pattern because 4 streams are well within its tracking capacity.

// New approach: flat twiddle loads (zero derivation)
W¹ ← load(tw_re[0*K+k])   // independent
W² ← load(tw_re[1*K+k])   // independent
W³ ← load(tw_re[2*K+k])   // independent
W⁴ ← load(tw_re[3*K+k])   // independent
//                            Total: 4 loads, ~4 cycle latency (parallel)

But there's a catch: at very high K (>2048), the twiddle table exceeds L2 cache. Four strided loads into L3 cost ~40 cycles each, while log3's single load costs ~40 once and derives the rest in registers. So the final design uses dual dispatch:

static inline void
radix5_tw_dit_kernel_fwd_avx2(..., size_t K) {
    if (K <= 2048)
        radix5_tw_dit_kernel_fwd_flat_avx2(...);    // 4 loads, 0 latency
    else
        radix5_tw_dit_kernel_fwd_log3_avx2(...);    // 1 load, 15cy latency
}

The threshold is a #define, tunable per hardware without regenerating code.

04 — AVX2 vs AVX-512 Strategy

The two ISAs have fundamentally different register budgets, requiring different scheduling strategies for the same algorithm.

Register pressure analysis — DIT twiddle kernel

05 — Why Not 2-Base?

We benchmarked a third variant: load W¹ and W², derive W³=W¹×W² and W⁴=(W²)² in parallel — only 2 loads, 5-cycle latency, with W³ and W⁴ independent.

StrategyLoadsDerivation opsLatencyCache misses (high K)
Log319 (3 cmul, serial)15 cycles1
2-base25 (1 cmul + 1 csq, parallel)5 cycles2
Flat400 cycles4

Benchmarks showed 2-base winning only at K=256 — a narrow sweet spot where L1 is just barely exceeded. Flat dominates everywhere below, log3 everywhere above. Two strategies cover the full range; the middle ground isn't worth the complexity.

06 — Planner: Reducing R=5 Exposure

The second half of the fix: don't put R=5 at high K in the first place. The old factorizer stacked R=5 stages, pushing the outermost to enormous K values. The fix: unlock R=25 at SIMD-aligned K, and prefer R=10 (fused 2×5) over separate R=2 + R=5.

NOld factorizationStagesNew factorizationStagesΔ
80032×5×5332×252−1 stage
400032×5×5×5432×25×53−1 stage
50008×5×5×5×5525×10×10×24−1 stage
1000016×5×5×5×5525×10×10×2×25R=10 fusion

Each eliminated stage saves one digit-reversal permutation + one buffer swap — roughly 20% overhead reduction per stage removed. The R=25 codelet handles K=25 efficiently (twiddle table = 24×25×16 = 9.6 KB, fits L1).

Key constraint: R=25 at K>1 only when K remains SIMD-aligned (K%4 == 0). Otherwise, decompose to R=5×5 which preserves alignment. This is enforced in the factorizer's greedy loop.

R=5 IL result: IL never wins for R=5. The 10→5 stream reduction doesn't overcome the interleave/deinterleave overhead. IL remains enabled in the registry (future codelet improvements may change this), but calibration suppresses it.

08 — Code Generation Architecture

One Python generator per ISA. Each produces 4 header files with identical public function names — drop-in replacement, dispatch unchanged.

Generated file matrix (per ISA)
  gen_r5_{isa}.py
    │
    ├─ fft_radix5_{isa}.h
    │    ├─ radix5_notw_dit_kernel_fwd_{isa}()
    │    ├─ radix5_notw_dit_kernel_bwd_{isa}()
    │    ├─ radix5_tw_dit_kernel_fwd_flat_{isa}()    ← internal
    │    ├─ radix5_tw_dit_kernel_fwd_log3_{isa}()    ← internal
    │    ├─ radix5_tw_dit_kernel_fwd_{isa}()          ← public (dispatches)
    │    └─ ... (bwd variants)
    │
    ├─ fft_radix5_{isa}_dif_tw.h
    │    └─ DIF variants (same dual pattern)
    │
    ├─ fft_radix5_{isa}_il.h
    │    └─ IL variants (notw + DIT tw, dual)
    │
    └─ fft_radix5_{isa}_il_dif_tw.h
         └─ IL DIF variants (dual)

  N1 IL (fft_radix5_{isa}_n1_mono_il.h) — genfft DAG, NOT generated

The public function signature is identical to the old log3 codelet. Dispatch headers call the same names. The flat/log3 selection happens inside the generated wrapper — invisible to the executor.

Three changes, stacking multiplicatively:

  1. Flat twiddle loads — eliminate 15-cycle serial derivation chain (10–28% codelet speedup)
  2. Dual dispatch — automatic fallback to log3 at K>2048 (no regression at high K)
  3. Planner R=25 unlock — fewer R=5 stages, each at lower K (~20% per eliminated stage)

Design principle: Measure, don't guess. Walk was killed by bench_walk. IL thresholds were set by bench_il. The flat/log3 crossover was found by bench_radix5_flat. No heuristics survived — every threshold in the R=5 path is backed by measurement.
