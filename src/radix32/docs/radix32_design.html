VectorFFT Architecture Reference

Radix-32 Codelet System

Three-tier twiddle strategy: flat CT at low K, binary ladder at mid K, pack+walk at high K. 4×8 Cooley-Tukey decomposition, AVX-512 + AVX2 + scalar. DIT, DIF, interleaved, and monolithic N1 paths.

31 codelet files 39,462 lines 7 generators 3 ISA tiers 3 twiddle strategies

01Three-Tier Twiddle Architecture

R=32 uses three different twiddle strategies, each optimal for a different K range. The dispatch selects automatically based on K and ISA.

Tier 1 — Flat

Load all 31 twiddle elements directly from the strided table. Simple, low overhead. Wins when the table fits in L1.

AVX-512: K ≤ 64  |  AVX2: all K  |  Scalar: all K

Tier 2 — Binary Ladder

Load 5 power-of-2 bases (W¹,W²,W⁴,W⁸,W¹⁶), derive all 31 via binary decomposition. 5 strided loads instead of 31.

AVX-512 only: 64 < K < 256 (U1)  |  K ≥ 256 (U2)

Tier 3 — Pack+Walk

Zero twiddle table. Pack data into contiguous blocks, compute twiddles on-the-fly from 5 walking accumulators. Eliminates both strided data and twiddle access.

Planner path: K > 512 (tw table > 32KB)
Why three tiers? At K=8 the twiddle table is 31×8×16=4KB — L1-resident, flat is cheapest. At K=128 it's 62KB — L1 spills, ladder's 5 loads save bandwidth. At K=2048 it's 1MB — even ladder's strided loads hurt, pack+walk converts all access to sequential.

04Binary Ladder Twiddle Derivation

The ladder loads 5 power-of-2 bases from the twiddle table and derives all 31 elements via binary decomposition. Since 31 = 2⁵ - 1, every n ∈ [1,31] is a unique 5-bit combination of {1, 2, 4, 8, 16}. The twiddle Wn is the product of bases corresponding to set bits. 

/* Load 5 bases from strided table (same layout as flat) */
b1  = load(tw[0*K+k])    // W¹   (bit 0)
b2  = load(tw[1*K+k])    // W²   (bit 1)
b4  = load(tw[3*K+k])    // W⁴   (bit 2)
b8  = load(tw[7*K+k])    // W⁸   (bit 3)
b16 = load(tw[15*K+k])   // W¹⁶  (bit 4)

/* Derive any W^n via binary tree (examples): */
W³  = b1 × b2             // 3  = 1+2
W⁵  = b1 × b4             // 5  = 1+4
W¹³ = b1 × b4 × b8        // 13 = 1+4+8
W³¹ = b1 × b2 × b4 × b8 × b16  // 31 = all bits

/* Average: ~2.5 cmuls per element (most are 2-3 bits) */
/* Total: ~80 cmuls to derive all 31 from 5 loads */

U1 vs U2 Unrolling

VariantK rangek-steps per iterationStrategy
U1 64 < K < 256 1 Load 5 bases, derive 31, apply to one k-block of 8 elements. Base loads amortized over 31 applications.
U2 K ≥ 256 2 Process two k-blocks per iteration. Bases loaded once, applied twice. Reduces base-load overhead by 50% and improves ILP: second block's butterflies overlap with first block's stores.
Ladder vs flat trade-off: Flat loads 31 twiddle pairs (62 strided loads) but zero derivation math. Ladder loads 5 pairs (10 strided loads) + ~80 cmuls. At K≥128 the 52 saved cache misses (stride 128×8=1KB per miss) outweigh 80 FMA instructions.

05Pack+Walk Engine

When K > 512, the twiddle table (31×K×16 bytes) exceeds L1 cache. The planner activates block-walk mode, which eliminates the twiddle table entirely and converts all strided data access to sequential.

Walk State

typedef struct {
    double step_re[5], step_im[5];       // scalar rotation per block advance
    double init_re[5][8], init_im[5][8]; // initial twiddle vectors
} radix32_walk_plan_t;                       // 120 doubles = 960 bytes

The walk state stores 5 base accumulators (W¹, W², W⁴, W⁸, W¹⁶), each as a SIMD vector of T=8 values. After each block, each accumulator is rotated by its step angle. The 31 twiddles per block are derived from these 5 accumulators using the same binary decomposition as the ladder.

Per-Block Pipeline

for (b = 0; b < K/T; b++) {
    /* 1. Pack: strided → contiguous (32×T doubles, sequential writes) */
    for (n = 0; n < 32; n++)
        copy T doubles from in[n*K + b*T] → block[n*T];

    /* 2. Kernel: flat codelet at K=T (all data + twiddles in L1) */
    radix32_tw_flat_dit_kernel_fwd_avx512(block, block_tw, T);

    /* 3. Unpack: contiguous → strided output */
    for (n = 0; n < 32; n++)
        copy T doubles from out_block[n*T] → out[n*K + b*T];

    /* 4. Walk: rotate 5 accumulators by step angle */
    for (i = 0; i < 5; i++)
        acc[i] = cmul(acc[i], step[i]);  // one cmul per base
}

064×8 Butterfly Structure

DFT-32 decomposes as 4 radix-8 DFTs followed by 24 internal twiddle multiplications (W32n) and then 8 radix-4 DFTs. The kernel uses 23 pre-computed W32 constants stored as static const double.

PropertyValue
Radix-8 butterflies4 (on rows of the 4×8 matrix)
Radix-4 butterflies8 (on columns)
Internal twiddle cmuls24 (W32 between row and column passes)
W32 constants23 static doubles (compiler broadcasts from memory)
External twiddle loads31 (flat) or 5 (ladder) per k-step
Data regs (peak)~14 YMM on AVX2 — fits 16 without split butterfly
Total FP ops (FMA variant)236 add + 62 mul + 198 FMA = 496 ops
Why no split butterfly for R=32? Unlike R=64 (which needs 16 data regs per radix-8 sub-butterfly × 8×8 structure = catastrophic pressure on 16 YMM), R=32's 4×8 structure peaks at ~14 YMM. With 23 W32 constants as memory operands (not hoisted), it fits naturally. The genfft DAG confirmed this: flat CT ties the DAG at every K for R=32.

11Design Decisions

Why three twiddle tiers instead of one?

A single strategy cannot be optimal across 3 orders of magnitude of K. Flat (31 direct loads) has zero derivation overhead and wins when the table fits in L1 (K≤64, table ≤ 32KB). Ladder (5 loads + 80 cmuls) trades compute for bandwidth when L1 spills (K=128-512). Pack+walk eliminates both strided data and twiddle access when even L2 is pressured (K>512), at the cost of pack/unpack memcpy.

Why is the ladder AVX-512 only?

The ladder derivation tree holds 5 base pairs (10 ZMM) + 2 accumulator pairs (4 ZMM) + 16 data registers + 2 constants = 32 ZMM. This exactly fills the AVX-512 register file. On AVX2 with 16 YMM, the 5 bases alone consume 10 registers, leaving only 6 for the butterfly — catastrophic spilling. AVX2 uses flat at all K; the planner's block-walk handles high-K on AVX2.

Why does the ladder use the same table as flat?

The flat twiddle table stores Wn at tw_re[(n-1)*K+k]. The ladder loads tw_re[0*K+k] (W¹), tw_re[1*K+k] (W²), tw_re[3*K+k] (W⁴), tw_re[7*K+k] (W⁸), tw_re[15*K+k] (W¹⁶) — these are just 5 specific rows of the same table. No separate ladder table is needed. The dispatch passes twr, twi as both flat and ladder arguments.

Why is the DAG not wired into dispatch?

Benchmarks showed flat CT ties or beats the genfft DAG at every K for R=32. The 4×8 CT structure naturally fits in registers (peak ~14 YMM) with no split-butterfly overhead. The DAG's 102 stack variables offer no advantage when the structured codelet already has zero spills. The DAG files are included as reference implementations and for the translate_genfft_tw.py generator to be validated against a second radix.

Why does R=32 not need R=64's split butterfly?

R=64's 8×8 CT has an inner radix-8 that needs 16 data regs simultaneously, competing with 14 log₃ bases and 3 constants = 33 total on 16 YMM. R=32's 4×8 CT has 8 data regs per radix-8 sub-pass + 10 flat twiddle loads (sequential, not persistent) + 2 constants = ~14 peak. It fits the 16-register budget with room to spare.
