# Bwd Direction — Sign Axis Added

## What changed

Bwd (inverse FFT) requires three changes from fwd:

1. **Internal CT twiddle sign flip.** The CT decomposition's INTERNAL twiddles `W^(n1·k2)` use `θ = -2π·n1·k2/N` for fwd, `θ = +2π·n1·k2/N` for bwd. This negates `sin(θ)` everywhere.

2. **External cmul becomes conjugate cmul.** Caller passes the SAME forward-convention W array in both directions (both store `cos`, `sin` of `-2π·j·k/(N·K)`). The bwd codelet conjugates internally:
   - Fwd: `x · W = (x_re·W_re - x_im·W_im, x_re·W_im + x_im·W_re)`
   - Bwd: `x · conj(W) = (x_re·W_re + x_im·W_im, -x_re·W_im + x_im·W_re)`

3. **Internal log3 derivations are unchanged.** When deriving `W^7 = W^3 · W^4` via cmul, both `W^3` and `W^4` are loaded from the array (with whatever convention is stored) and their product is `W^(3+4)` regardless of sign. The cmul formula stays in fwd form for these.

## Implementation

`lib/dft.ml`:
- New polymorphic-variant type `[`Fwd | `Bwd]` (kept separate from the `direction = DIT | DIF` axis — they're orthogonal).
- `dft`, `dft_ct`, `dft_direct` all take `?(sign = `Fwd)`. The CT twiddle code multiplies `theta` by `sgn` (where `sgn = -1.0` for Fwd, `+1.0` for Bwd).
- `cmul_pattern` takes `?(conj = false)`. Internal log3 derivations call without `~conj`. External pre-multiply (DIT) and post-multiply (DIF) call with `~conj:(sign = `Bwd)`.
- `dft_expand_twiddled` and `dft_expand_twiddled_spill` thread `~sign` through.

`bin/gen_radix.ml`:
- New `--bwd` flag.
- Function name picks up `bwd` suffix instead of `fwd`.

The recipe (Spill + SU + cluster-sequential PASS 2) and log3 both work unchanged — they're sign-agnostic. Auto-recipe applies the same way.

## Validation

**Byte-identical regression check on fwd.** Every fwd codelet generated after the change is bit-for-bit identical to the version before. The sign axis defaults to Fwd, so existing behavior is preserved.

## Note on hand bwd vs fwd timing

Hand's bwd codelet at R=64 is itself ~30% slower than hand's fwd codelet at small K (e.g., K=64: 4250 ns fwd vs 5550 ns bwd). This makes our wins look smaller in absolute LR/H terms — we beat fwd by 37% at K=64 (LR/H=0.63) but bwd by only 16% (LR/H=0.84) because the bwd baseline is slower to begin with. Our bwd ABSOLUTE time at K=64 is ~4700 ns, comparable to our fwd time of ~4500 ns. The conjugate cmul pattern is the same op count as the regular cmul, just different signs, so absolute perf is similar.

## Final variant matrix

The variant axis has now grown to:

```
{R=4, R=8, R=16, R=32, R=64} × {DIT, DIF} × {Fwd, Bwd} × {flat, log3} × {t1, t1s} × {AVX-512, AVX2}
```

That's 5 × 2 × 2 × 2 × 2 × 2 = 320 functional combinations, all generated by the same generator from the same math layer. The recipe and log3 apply orthogonally across all axes.

What remains:
- AVX2 + Bwd (likely fine; the changes are ISA-agnostic, but not measured)
- t1s + Bwd (same — likely fine, not measured)
- DIF + log3 + Bwd (should also work via the same path)

Each of these is `dune exec bin/gen_radix.exe -- --twiddled --emit-c <flags>` away.
