# 45. Regression Bench — Hand vs OCaml at R=16, 25, 32, 64

## Motivation

After completing the twidsq math layer (doc 43) and emitter stride
parameterization (doc 44), check whether any of the recent edits
accidentally regressed performance on the normal (non-twidsq) t1_dit
codelets that production cascades depend on.

## Setup

For each R ∈ {16, 25, 32, 64}:
- **Hand**: codelet generated by `gen_radix{R}.py --variant ct_t1_dit`
- **OCaml**: codelet generated by `gen_radix.exe {R} --twiddled --in-place`
  with **default production config** (recipe + SU + spill auto-fires
  for R≥5)

Both compiled identically with `gcc-11 -O3 -mavx512f -mavx512dq -mfma
-march=skylake-avx512 -flive-range-shrinkage`.

Bench harness: 100-iter warmup, best of 7 trials of 5000 iterations,
in-place codelet with `ios = me = K` and aligned 64-byte buffers.

5 full runs of the bench, reporting the median ratio and the
[min, max] spread to disambiguate noise from real differences.

## Interpretation

**No regressions from recent edits.** The emitter changes in docs 43-44
only affect the twidsq path (gated on `twidsq=true`); the non-twidsq
t1_dit emission walks the same code paths as before. The numbers
above reflect the steady-state state of the codebase, not a regression
window.

**R=16 dominates hand by 11-17%** across the full K sweep. This is
the strongest result — our N1=4, N2=4 Cooley-Tukey decomposition with
the recipe (SU + spill) produces measurably better code than the
hand-coded variant. The win scales with K: at K=64 it's TIE, at
K≥128 it's a solid double-digit win.

**R=32 and R=64 are at parity with hand.** Median ratios sit between
0.97 and 1.01 across all K. The hand-coded variants are well-tuned
at these sizes; we match them within noise.

**R=25 is the weakest case** — TIE at small K but ~7-14% slower at
K=256 and K=1024. R=25 = 5×5 CT decomposition. The 5-point sub-DFT
is direct (prime), so the hand-coded version's R=5 inner butterflies
were probably hand-tuned with the Winograd-style constants visible
in `r25_hand.h` (`cA, cB, cC, cD`). Our CT-decomposed R=25 currently
inherits the same Direct R=5 conjugate-pair construction from doc 23
(which is good standalone — R=5 prime correctness passes) but doesn't
specifically tune for the cascaded N=25 = 5×5 case.

This isn't a recent regression — it's a pre-existing gap that the
recent edits don't affect. The R=25 gap is the natural next investigation
target if composite-with-prime-factors codelets matter for production
use; for now most cascades terminate at pow2 radixes which are at
parity or better.

## Bench noise level on the container

Spread between min and max across 5 runs:
- Most configurations: 0.04-0.10 (4-10% noise)
- Larger K with cache effects: up to 0.20 (R=25 K=256)

This is expected for a containerized environment without isolated CPU.
On an i9-14900K with `taskset -c 0` and `cpupower` frequency pinning,
spread should drop to <0.03 (<3%). The qualitative conclusions (R=16
WINS, R=32/64 TIE, R=25 marginal slow) hold.

## What's still on the table

The R=25 gap is the only specific performance concern. Potential
improvements if it ever matters:

1. **Recognize composite-with-prime-factors structure**: when N=p×q
   with p prime ≥3, the inner p-point DFTs share the conjugate-pair
   construction. Currently each gets emitted independently; sharing
   subsum patterns across the q parallel p-point DFTs might recover
   the gap. Not free — would need a generator pass that looks across
   sub-DFTs.

2. **Try the t1s variant for R=25**: the hand-coded version above is
   the t1 variant (vector twiddles). For R=25 the t1s variant
   (scalar-broadcast twiddles) might be a better default given the
   smaller twiddle table count. Worth a separate bench.

3. **Different scheduler choice**: the default recipe (SU+spill+GH-on-AVX2)
   is tuned for pow2 codelets where it shines (R=16 results show this).
   For composites, the SU heuristic might over-spill; trying `--no-spill`
   or `--bb` on R=25 specifically could close the gap.

Not priority right now since the user's production targets are pow2
in cascades plus pure-prime codelets — both of which are at parity
or better with hand. R=25 is a generic composite that's neither.
