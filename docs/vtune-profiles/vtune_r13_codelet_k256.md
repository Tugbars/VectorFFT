# VTune Profile: R=13 Isolated Codelet (t1_dit_fwd, K=256)

**Date:** 2026-04-13  
**CPU:** Intel i9-14900KF (Raptor Lake), P-core only, 5.69 GHz turbo  
**Binary:** `vfft_bench_codelet.exe 256 5000 13 t1_dit_fwd`  
**Codelet:** `radix13_t1_dit_fwd_avx2` — prime-radix DFT-13 (genfft Winograd-style with Sethi-Ullman scheduling), single stage, AVX2  
**Data size:** R=13, K=256, total=3328 elements = 52 KB (re+im), fits L1  
**Analysis:** Microarchitecture Exploration (uarch-exploration)

---

## Baseline performance

| Metric | Value |
|---|---|
| ns/call | 1766 |
| CPE (cycles / (R × K)) | **2.97** |
| GFLOP/s | 25 |
| Retiring | **59.7%** of pipeline slots |
| CPI Rate | 0.278 |

R=13 is a **prime-radix codelet** generated by genfft and post-processed
with Sethi-Ullman scheduling for minimal register usage. Like R=11, the
DFT-13 butterfly cannot decompose — it's a single-stage Winograd-style
implementation with the dependency tree height already optimized.

| — DTLB Store Overhead | 11.5% of store-cycles |

Key observations:

- **L1 Latency Dependency 41.6%** — only slightly less than R=11's 43.0%.
  The DFT-13 FMA dependency chain is just as long as DFT-11's. Sethi-Ullman
  scheduling has already minimized this — it's the algorithmic ceiling.
- **Memory Bound 6.0%** — among the lowest we've measured. Single-stage
  execution + small working set means almost no cache pressure.
- **Store Bound 0.1%** — effectively zero, like R=11 and R=12.
- **DTLB Store Overhead 11.5% of store-cycles**, but Store Bound is only
  0.1% total — the absolute cost is negligible.
- **Split Stores 2.6%** — much lower than R=11's 16.4%. The 13-row output
  layout happens to align better at K=256 stride than R=11's 11-row layout.

## Bad Speculation — back to baseline

| Source | % of Pipeline Slots |
|---|---|
| Branch Mispredict | 0.2% |
| Machine Clears | 0.2% |

**0.4% total, normal background level**. Confirms the 4.4% Machine Clears
seen in R=11 is specific to that codelet — NOT a generic prime-radix
Winograd issue.

What this tells us:

- Sethi-Ullman + genfft is not inherently problematic for speculation
- Whatever pattern in R=11 triggers machine clears is unique to that
  specific codelet (instruction sequence, register allocation, or some
  edge-case memory pattern)
- R=13, R=17, R=19, R=23 (other genfft primes) need to be tested
  individually — we cannot generalize from R=11

**Port 1 dominates over Port 0** (65.7% vs 55.9%) — interesting asymmetry.
Compared with R=11 which was balanced (62.6 / 64.4), R=13's butterfly
generates more uops that prefer Port 1 (FMA only). Likely a side-effect of
how genfft schedules the DFT-13 multiplications.

**Store Op Utilization 34.7%** is the highest of all small composites and
primes profiled — DFT-13 produces 13 output values per call, more than
any other codelet in this group, and they all need to be stored.

| Radix | CPE | Retiring | Memory Bound | Bad Spec | Dominant bottleneck |

R=13 is the **cleanest example of the prime-radix Winograd profile**:

- High retiring (59.7%) — almost as good as R=10 despite being prime
- Lowest Memory Bound across all small composites and primes (6.0%)
- Long FMA chain (41.6% L1 Latency Dep) — algorithmic ceiling
- No speculation pathology (unlike R=11)
- Higher per-element CPE than smaller primes (2.97 > R=11's 2.70)
  because DFT-13 has more multiplications

## Optimization analysis (no changes applied)

### Deferred constants — not applicable (single-stage)

R=13 is single-stage. There is no PASS 2 to defer constants into.
Same as R=11.

### Twiddle prefetch — predicted to regress

L1 Bound 0.8% — no load latency to hide. Same regime as R=8/R=10/R=11/
R=12/R=20/R=25.

### `PREFETCHW` on outputs — not applicable

Store Bound 0.1%. Nothing to optimize.

### No machine-clears investigation needed

R=13 doesn't have the speculation issue R=11 has. The Sethi-Ullman +
genfft toolchain produces clean code for DFT-13.

---

## Final conclusion

R=13 on AVX2 is **near its achievable ceiling** for this microarchitecture.
The Sethi-Ullman-scheduled genfft Winograd butterfly minimizes dependency
tree height; the remaining ~42% L1 Latency Dependency is intrinsic to the
DFT-13 algorithm — not fixable at the instruction level.

R=13 is also a useful **negative control for the R=11 anomaly**:
- Same toolchain (genfft + Sethi-Ullman)
- Same prime-radix structure
- Same single-stage execution
- Same low memory pressure
- **Different speculation behavior** (0.4% vs 4.6%)

Whatever causes R=11's 4.4% machine clears is **specific to the R=11
codelet**, not a generic property of prime-radix Winograd codelets.

**Baseline locked (unchanged)**: `t1_dit_fwd` 1766 ns / CPE 2.97.

---

## Heuristic reinforced

Prefetch heuristic still holds: L1 Bound 0.8% means no load latency to
hide. R=13 joins R=10, R=11, R=12, R=20, R=25 in the "no software
optimization room available on Raptor Lake" category.

The full prime-radix + small-composite cluster is now characterized:

| Type | Retiring | Memory Bound | L1 Latency Dep | Action |
|---|---|---|---|---|
| Composite (R=10/12/20/25) | 50–63% | 8–20% | 30–39% | None — at ceiling |
| Prime + Sethi-Ullman (R=13) | 60% | 6% | 42% | None — at ceiling |
| Prime with anomaly (R=11) | 58% | 6% | 43% | **Investigate machine clears** |

Future work: R=11 machine-clears investigation. May or may not yield
recoverable performance; either way, characterizing the cause helps
understand whether other genfft outputs are at risk.
