Radix-64 Codelet System

Multi-ISA DFT-64 codelets: genfft DAG at low K, 8×8 Cooley-Tukey with log₃ twiddle derivation at high K, scalar fallback for any K. Split re/im layout, AVX2 + AVX-512 + scalar.

01Dispatch Overview

Every call enters fft_radix64_dispatch.h which selects the optimal codelet based on K (number of parallel DFT-64 problems), direction (fwd/bwd), and available ISA. The dispatch is a waterfall: AVX-512 → AVX2 → scalar.

02The Two Engines

Engine A — genfft DAG DAG

Mathematical schedule from FFTW's genfft compiler (Frigo & Johnson, 1999/2005). The DAG is a single fused operation graph that interleaves twiddle multiplications and butterfly additions to minimize temporary variables. Our translate_genfft_tw.py parses the scalar FMA codelet from FFTW's t1_64.c and emits vectorized split-format code with native intrinsics.

PropertyValue
FMA ops per k-step398 (259 FMA + 259 FNMS from genfft optimizer)
ADD/SUB/MUL ops638
Data loads128 (64 re + 64 im)
Twiddle loads126 (63 re + 63 im, strided)
Stores128
Stack variables~190 (SSA-style, each assigned once)
Constants15 (broadcast from memory)
Spill bufferNone — fused schedule
BackwardPointer-swap trick: fwd(im,re) = bwd(re,im)

Strength: Minimum arithmetic operations. The DAG has zero intermediate spill — every value feeds directly into its consumer. At low K the twiddle table fits in L1, so the 126 strided loads are fast.

Weakness: 190 live variables on 32 ZMM → compiler spills ~160 to stack. At high K, 126 strided twiddle loads (stride K×8 bytes) thrash L1. Both effects grow with K.

Engine B — 8×8 Cooley-Tukey + log₃ 8×8CT

Decompose DFT-64 as 8×8: first 8 radix-8 DFTs on rows, apply W64 internal twiddles, then 8 radix-8 DFTs on columns. The 63 external twiddle bases are derived from just 2 table loads via a depth-3 multiplication tree (log₃).

PropertyValue
Radix-8 butterflies16 (8 row + 8 column)
Internal twiddle cmuls49 (W64 from static arrays)
External twiddle loads2 per k-step (W¹, W⁸ from table)
External twiddle derivation12 cmuls (log₃ tree → 14 bases)
Total FP ops~688 (more than DAG)
Data regs (peak)16 (8 re + 8 im per radix-8)
Spill bufferAVX2: 128 stores + 128 loads (split butterfly)
AVX-512: 0 (fits in 32 ZMM)
BackwardSeparate bwd kernel (conjugate twiddles)

Strength: Only 2 strided twiddle loads per k-step. Data access is structured: 8 elements at a time, predictable stride. Register reuse across 16 butterflies keeps pressure bounded. Scales gracefully to any K.

Weakness: 36% more arithmetic ops than the DAG. The 8×8 transpose (even/odd split for AVX2) adds latency.

The K=12 crossover: Benchmarks show the DAG wins at K≤8 (less setup overhead, twiddles L1-resident), while 8×8CT wins at K≥16 (2 loads vs 126, structured access pattern). The threshold RADIX64_DAG_K_THRESHOLD=12 is configurable.

Scalar

Both N1 and DAG tw codelets have scalar variants generated by the same Python translators with isa='scalar'. The scalar DAG is the only codelet that handles arbitrary K values (K=1, K=7, K=13, etc.) — it's the universal fallback. The N1 scalar codelet uses the same 8×8 CT structure as SIMD but with double instead of vectors.

04Log₃ Twiddle Derivation

External twiddles Wn for n=1..63 are organized as an 8×8 grid: n = 8·row + col. Only W¹ and W⁸ are loaded from the table per k-step. All 14 bases are derived in-register: 

/* Load from table: 2 strided loads per k-step */
ew1 = load(tw_re[0*K+k])   // W¹
ew8 = load(tw_re[7*K+k])   // W⁸

/* Column bases: W² .. W⁷ (6 cmuls) */
ew2 = ew1 × ew1             // W² = (W¹)²
ew3 = ew1 × ew2             // W³
ew4 = ew2 × ew2             // W⁴ = (W²)²
ew5 = ew1 × ew4             // W⁵
ew6 = ew3 × ew3             // W⁶ = (W³)²
ew7 = ew3 × ew4             // W⁷

/* Row bases: W¹⁶ .. W⁵⁶ (6 cmuls) */
ew16 = ew8 × ew8            // W¹⁶ = (W⁸)²
ew24 = ew8 × ew16           // W²⁴
ew32 = ew16 × ew16          // W³² = (W¹⁶)²
ew40 = ew8 × ew32           // W⁴⁰
ew48 = ew24 × ew24          // W⁴⁸ = (W²⁴)²
ew56 = ew24 × ew32          // W⁵⁶

/* Apply: W^n = W^col × W^(8·row) — 2 cmuls for general n */
/* Special: n%8==0 → 1 cmul, n<8 → 1 cmul, n==0 → free */

Cost: 2 table loads + 12 cmuls (derivation) + up to 2 cmuls per element (application). For the 56 non-trivial twiddle elements: ~112 cmuls total.

Benefit: 2 strided memory accesses instead of 126. At K=2048 this is the difference between 2×16KB stride and 126×16KB stride — the L1 prefetcher handles 2 streams easily.

05Internal W64 Twiddles

The 8×8 CT applies W64n (n=0..63) between the row and column radix-8 passes. These are compile-time constants stored in static arrays: 

static const double iw_re[64] = { 1.0, 0.9952..., ... };
static const double iw_im[64] = { 0.0, -0.0980..., ... };

/* Applied as broadcast: */
__m256d wr = _mm256_set1_pd(iw_re[e]);   // vbroadcastsd from memory
__m256d wi = _mm256_set1_pd(iw_im[e]);   // no register hoisting needed

Previous approach hoisted 44 const __m256d declarations (63 twiddle pairs) at function scope — 88 YMM live simultaneously on 16 registers. The static-array approach uses zero persistent registers: each broadcast is a single-instruction memory operand that the compiler can schedule independently.

Special cases are handled without multiplication: W64⁰ = 1 (skip), W64¹⁶ = −j (swap+negate), W64³² = −1 (negate), W64⁴⁸ = +j (swap), W64⁸ and W64²⁴ use the √2/2 constants already in registers (vc, vnc).

10Design Decisions

Why not DAG everywhere?

The genfft DAG has optimal operation count but 190 stack variables and 126 strided twiddle loads. At K≥16 these costs dominate: the 8×8CT's structured access pattern and register reuse across 16 butterflies wins by 10–30%. The DAG remains optimal at K≤8 where twiddles fit in L1 and setup cost matters.

Why not 8×8CT everywhere?

At K=8 the 8×8CT's log₃ derivation (12 cmuls) and spill buffer setup (AVX2) have fixed overhead that the DAG avoids. The DAG's fused schedule processes all 64 elements with zero setup latency.

Why split butterfly on AVX2?

A monolithic radix-8 butterfly needs 16 data registers simultaneously (8 re + 8 im). With log₃ bases (28 YMM dormant but in scope) and 3 constants, that's 47 YMM on 16 hardware registers — catastrophic spilling in the innermost loop. The split butterfly processes even/odd halves separately, capping peak live at 12 YMM. The compiler spills log₃ bases to sequential stack (not strided), which the prefetcher handles cheaply.

Why pointer-swap for backward DAG?

DFT backward = conj(DFT forward(conj(x))). For split re/im: swapping the re↔im pointers is equivalent to conjugation. This gives us the backward codelet for free — zero additional generated code, zero maintenance burden, mathematically exact.

Why scalar DAG as the universal fallback?

The planner may produce K values that aren't multiples of 4 or 8 (e.g. N=448 = 7×64 → stage 1 has R=64, K=7). Neither AVX2 nor AVX-512 can handle K=7. The scalar DAG processes one element per k-step and works for any K, including K=1 (N1 equivalent). This eliminates all "void fallback" paths that previously produced garbage output.
