Why the Transpose Exists

A visual walkthrough of what permutation and transpose solve in the old FFT architecture, why they cost 30–40% of execution time, and how a single signature change eliminates both.

01 The Setup

We want to compute a DFT of size N = 8, factored as R × M = 2 × 4 using decimation-in-time (DIT). The input is 8 complex values in natural order.

DIT says: split the input into R = 2 groups by taking every R-th element. Group 0 gets the even-indexed elements. Group 1 gets the odd-indexed elements. Then compute an M-point DFT on each group independently, apply twiddle factors, and combine with an R-point butterfly.

Group 0 needs elements at positions 0, 2, 4, 6 — stride 2 apart.
Group 1 needs elements at positions 1, 3, 5, 7 — also stride 2 apart.
Neither group is contiguous in memory.

02 The Old Codelet's Limitation

Our codelet has this signature:

void radix4_fwd(const double *in_re, const double *in_im,
                 double *out_re, double *out_im,
                 const double *tw_re, const double *tw_im,
                 size_t K);

// Reads: in_re[j * K + k] for j = 0..3, k = 0..K-1
// Writes: out_re[j * K + k] for j = 0..3, k = 0..K-1
//
// Same K for both input and output.
// Cannot read at stride 2 and write at stride 1.

The single K parameter is used for both the input element spacing and the output element spacing. If K = 2, it reads at stride 2 and writes at stride 2. If K = 1, it reads at stride 1 and writes at stride 1. It cannot read at stride 2 (to gather the even elements) and write at stride 1 (to pack them contiguously). That asymmetry is exactly what DIT requires.

The Root Cause

DIT decomposition needs the inner DFT to read at stride R (gather every R-th element) and write at stride 1 (pack results contiguously for the next stage). One parameter K cannot express two different strides. The transpose exists to perform this stride change outside the codelet.

Total cost

04 Why It's a Fundamental Limitation

This isn't a bug or a lazy implementation. It's a necessary consequence of the codelet interface. Let me show why with the actual memory access pattern:

// The inner DFT-4 codelet needs to:
// READ x₀, x₂, x₄, x₆ (stride 2 in the input)
// WRITE G₀₀, G₀₁, G₀₂, G₀₃ (stride 1 in the output)

// But the codelet computes:
  in_re[j * K + k] → out_re[j * K + k]

// If K = 2: reads at stride 2 ✓, writes at stride 2 ✗
// If K = 1: reads at stride 1 ✗, writes at stride 1 ✓
// No value of K satisfies both.

The problem scales with every level of recursion. For N = R₁ × R₂ × R₃, there are two stride changes (between levels 1→2 and 2→3), each requiring a transpose. More levels = more transposes = more wasted bandwidth.

Our actual benchmarks measured this exactly:

// N = 2048, 2-level FFT: 32 × 64
  Inner DFT-64 codelet:  3374 ns (44%)
  SIMD 4×4 Transpose:   1368 ns (18%) ← pure waste
  Outer DFT-32 codelet:  2882 ns (38%)
  Total:                7623 ns
  FFTW MEASURE:         5001 ns

// Without the transpose: 3374 + 2882 = 6256 ns
// That would beat FFTW. The transpose is the entire gap.

05 The New Signature

Replace the single K with five parameters that decouple input stride from output stride:

void radix4_n1_fwd(
    const double *ir, const double *ii,
    double *or_, double *oi,
    size_t is,    // input element stride
    size_t os,    // output element stride
    size_t vl,    // batch count (vectorization length)
    size_t ivs,   // input batch stride
    size_t ovs);  // output batch stride

// Reads: ir[j * is + k * ivs]
// Writes: or_[j * os + k * ovs]

// is and os are independent. The codelet can now
// read at stride 2 and write at stride 1.

The old codelet is a special case: is = os = K, ivs = ovs = 1. The new signature is strictly more general — it can do everything the old one did, plus the stride change that eliminates the transpose.

07 Side-by-Side Comparison

Old: K conflates all strides

Pass 1: Transpose (N reads+writes)
Pass 2: Inner DFT-M, K=1
Pass 3: Transpose (N reads+writes)
Pass 4: Outer DFT-R, K=M

4 passes. 2 are pure data movement.

Memory traffic: 4 × 2N = 8N

Transposes touch every element twice, polluting cache and evicting the data the next codelet needs.

New: is ≠ os eliminates transpose

Pass 1: Inner DFT-M, is=R, os=1
Pass 2: Twiddle + DFT-R, in-place

2 passes. Both do useful arithmetic.

Memory traffic: 2 × 2N = 4N

Half the memory bandwidth. Data written by pass 1 is still in L1 when pass 2 reads it.

08 At Scale: 3 Levels

For N = 4096 = 4 × 16 × 64, the old architecture needs a transpose between every level — four extra passes of pure data movement. The new architecture needs zero.

4. 64× DFT-16 (K=1)

New: 3 passes (all useful)

1. Inner DFT-64, is=64, os=1
2. Twiddle + DFT-16, in-place

Memory: 6 × N = 24576 element accesses.
Every access carries useful arithmetic.
Sub-problems fit in L1 at each level.

The Recursion

The recursive executor passes strides down the tree. Level 1 calls the inner DFT with is = R₁ × parent_is (decimated input) and os = parent_os (contiguous output). The stride change propagates through the recursion automatically. No level ever needs to see the full array layout — it just processes its local sub-problem with the strides it was given. The permutation that DIT requires is distributed across the codelet load addresses instead of materialized as a separate pass.

09 What Changes in the Code

The butterfly math inside the codelet stays identical. Only two things change:

1. Load addresses

// Old:
__m256d x0 = _mm256_load_pd(&in_re[0 * K + k]);
__m256d x1 = _mm256_load_pd(&in_re[1 * K + k]);

// New:
__m256d x0 = _mm256_load_pd(&ir[0 * is + k * ivs]);
__m256d x1 = _mm256_load_pd(&ir[1 * is + k * ivs]);

2. Store addresses

// Old:
_mm256_store_pd(&out_re[0 * K + k], result0);
_mm256_store_pd(&out_re[1 * K + k], result1);

// New:
_mm256_store_pd(&or_[0 * os + k * ovs], result0);
_mm256_store_pd(&or_[1 * os + k * ovs], result1);

The arithmetic between the loads and stores — every FMA, every add, every subtract — is completely unchanged. The genfft DAG, the Sethi-Ullman scheduling, the ILP interleaving — all of it stays exactly as-is. The entire optimization is in the addressing, not the math.

Summary

The transpose and permutation exist because the old codelet interface uses one parameter for both input and output stride. DIT decomposition requires different strides at each level. The mismatch forces explicit data rearrangement passes that consume 30–40% of execution time and pollute the cache hierarchy. Splitting K into separate is/os parameters lets the codelet perform the stride change as part of its normal load/store operations, eliminating every transpose and permutation from the FFT execution.
