# 02 — Codelet taxonomy

Every codelet variant the wisdom system has to choose between. Three
levels of granularity stack: **variant** (fine-grained codelet) → **dispatcher**
(family of variants sharing a calling convention) → **protocol** (twiddle
layout family).

The wisdom file records per-stage choices at the **protocol** level
(four codes: `FLAT`, `LOG3`, `T1S`, `BUF`). The within-protocol selection
of the specific dispatcher / sub-variant happens internally at runtime
(the dispatcher's `(me, ios)` selector picks the best variant from its
family). So when wisdom says "use FLAT at this stage," the registry's
flat codelet pointer dispatches to whichever sub-variant is best for
the runtime `(me, ios)`.

## Level 1 — Variant (the codelet itself)

A *variant* is one specific codelet C function emitted by the codelet
generator. At R=16 alone, the bench harness measures these:

| Variant | Brief description |
|---------|-------------------|
| `ct_t1_dit` | Baseline DIT-flat — vector twiddle loads at full per-K width |
| `ct_t1_dif` | Baseline DIF-flat — pre-butterfly twiddle |
| `ct_t1s_dit` | Scalar-broadcast twiddle: one scalar per leg per stage, broadcast across K |
| `ct_t1_dit_log3` | DIT log3 — derives twiddles via cmul chains from a small base set |
| `ct_t1_dif_log3` | DIF log3 |
| `ct_t1_dit_log3_isub2` | AVX-512 variant of log3 — useful on AVX-512, **not on AVX2** |
| `ct_t1_dit_log_half` | AVX-512 variant of log3 — same situation |
| `ct_t1_buf_dit_tile64_temporal` | Buffered DIT-flat with 64-element tile and temporal stores |
| `ct_t1_buf_dit_tile128_temporal` | Same, 128-element tile |

Variant naming pattern: `ct_t1{s?}_{dit|dif}{_logN?}_{buf?}_{tile_options?}`.
The `ct_` prefix is historical (Cooley-Tukey).

R=16 has 9 variants. R=32/64 have similar counts (buffered tile sizes
multiply). R=4/8 have fewer because they're smaller codelets without
buffered or scalar-broadcast variants.

## Level 2 — Dispatcher (family of variants)

A *dispatcher* is a per-(me, ios) selector function over multiple
variants in the same family. Generated by `select_and_emit.py` after
the bench reduces variants to one winner per (me, ios). Examples:

| Dispatcher | Variants it selects from |
|-----------|--------------------------|
| `t1_dit` | `ct_t1_dit`, plus any log/u2 sub-variants of flat |
| `t1_dit_log3` | `ct_t1_dit_log3`, `ct_t1_dit_log3_isub2`, `ct_t1_dit_log_half` |
| `t1s_dit` | `ct_t1s_dit` (only one variant in the family) |
| `t1_buf_dit` | `ct_t1_buf_dit_tile64_temporal`, `ct_t1_buf_dit_tile128_temporal` |

The registry refers to dispatchers, not variants. When the planner picks
"flat" for a stage, the dispatcher inside `t1_dit` picks one specific
variant based on (me, ios) — the registry doesn't know which.

## Level 3 — Protocol (twiddle layout family)

A *protocol* is the high-level twiddle-layout strategy. Four protocols
correspond directly to the wisdom file's per-stage variant codes:

| Protocol | `vfft_variant_t` code | Characteristic | Memory traffic |
|----------|----------------------|----------------|-----------------|
| **flat** | `VFFT_VAR_FLAT = 0` | Full (R-1)·K twiddle table loaded as vectors | High |
| **t1s**  | `VFFT_VAR_T1S = 2`  | Scalar twiddles broadcast across K | None for twiddles |
| **log3** | `VFFT_VAR_LOG3 = 1` | Derive twiddles via cmul chains from a small base set | Minimal |
| **buf**  | `VFFT_VAR_BUF = 3`  | Buffered flat with output tile staging | Same as flat, better access pattern |

The calibrator's variant Cartesian iterator (`vfft_variant_iter_*`)
walks all 4 protocols per stage in DIT (only FLAT and LOG3 in DIF —
see [07_dif_filter.md](07_dif_filter.md)) and benches each. The
winning protocol's code is what gets written to the wisdom file's
`variant_codes[]` array.

## Orientation: DIT vs DIF (an orthogonal axis)

Each variant comes in DIT and DIF flavours:

- **DIT** (Decimation In Time): twiddle multiplications happen *before*
  the butterfly. Forward executor structure default.
- **DIF** (Decimation In Frequency): twiddle multiplications happen
  *after* the butterfly. Backward executor structure default.

DIT and DIF compute different output buffers from the same input + the
same twiddles, so they are **not interchangeable per stage**. A plan
chooses one orientation for the whole forward pass via the
`use_dif_forward` flag.

T1S and BUF protocols only have DIT variants — no DIF analogs exist.
LOG3 has both. FLAT has both.

See [07_dif_filter.md](07_dif_filter.md) for why the wisdom system filters
DIF variants out of per-stage selection.

## Within-protocol dispatcher selection

The calibrator's variant code is at the protocol level (one of four).
But within FLAT there are actually two dispatchers — `t1_dit` (the
default) and `t1_buf_dit` (buffered). The wisdom file's `BUF` variant
code is reserved for the case where the buffered flat dispatcher
explicitly wins; if `FLAT` is recorded, the runtime dispatcher's
`(me, ios)` selector inside `t1_dit` picks the best sub-variant.

Within LOG3 there are also multiple sub-variants (`ct_t1_dit_log3`,
`ct_t1_dit_log3_isub2`, `ct_t1_dit_log_half`) that the dispatcher
selects from at runtime. The wisdom code says "LOG3" — the dispatcher
picks which LOG3 sub-variant.

So plan-level wisdom records the *protocol* the calibrator measured to
win; sub-variant selection within that protocol is a runtime concern
of the dispatcher.

## Per-radix variant inventory

Not every radix has every variant. From the registry's
`vfft_stage_variants`:

| Radix | n1 | t1 (flat) | t1s | log3 | buf | t1_dif | t1_dif_log3 |
|-------|----|----|------|------|-----|--------|-------------|
| 2     | ✓ | ✓ | — | — | — | ✓ | — |
| 3, 5, 6, 7, 10, 11, 12, 13, 17, 19, 20, 25 | ✓ | ✓ | ✓ | ✓ | — | — | — |
| 4, 8 | ✓ | ✓ | ✓ | ✓ | — | ✓ | — |
| 16, 32, 64 | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |

Highlights:

- **R=2 is the bootstrap radix** — only flat. No tuning, no t1s, no log3. The
  wisdom dispatcher's default-zero handles the missing predicates.
- **DIF orientation is pow2-only** (R=2/4/8/16/32/64). Other radixes can
  participate in a forward DIT plan, but a DIF plan must use only pow2
  factors.
- **BUF is R=16/32/64-only.** The buffered output staging only pays off
  for codelets large enough to amortize the tile-loop overhead.
- **DIF-LOG3 is R=16/32/64-only.** Other radixes' log3 variants weren't
  emitted in DIF orientation because the codelet generator hadn't been
  extended at v1.0 freeze.

## ISA-specific variant subset

Some variants are AVX-512-targeted and rarely (or never) win on AVX2:

- `ct_t1_dit_log3_isub2` and `ct_t1_dit_log_half` are AVX-512 log3 variants.
  They're benched on AVX2 hosts (the bench harness times every variant
  the codelet generator emits), but their wins on AVX2 are negligible —
  the AVX-512 tuning leaves them sub-optimal at AVX2's narrower lanes.

The AVX2 wisdom emitted on a Raptor Lake host should not show
`isub2` / `log_half` as winners; if it does, that's a calibration noise
artifact worth investigating.

## Variant codes in the wisdom file

The plan-level wisdom file (`vfft_wisdom_tuned.txt`) uses integer codes
for the 4 protocols, not variant names:

```
v=0:FLAT   v=1:LOG3   v=2:T1S   v=3:BUF
```

These map to `vfft_variant_t` in `registry.h`:

```c
typedef enum {
    VFFT_VAR_FLAT = 0,
    VFFT_VAR_LOG3 = 1,
    VFFT_VAR_T1S  = 2,
    VFFT_VAR_BUF  = 3,
    VFFT_VAR_COUNT = 4,
} vfft_variant_t;
```

Within-flat dispatcher selection (`t1_dit` vs `t1_buf_dit`) is *not*
encoded at this level — the variant code says only "use flat protocol"
and the dispatcher picks the right flat sub-variant at runtime via its
internal (me, ios) selector. So `BUF` as a wisdom-level variant code
means "the buffered-flat dispatcher won this stage outright" — a stronger
claim than just "flat won and might pick buf via dispatcher."
