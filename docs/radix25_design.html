VectorFFT Architecture Reference

Radix-25 Codelet System

Non-power-of-2 workhorse for N with factors of 5. 5×5 Cooley-Tukey decomposition, log₃ twiddle derivation, full scalar fallback for odd K values. The first radix where the planner's block-walk engine activates in production (K=200 from N=5000).

41 codelet files ~28K lines 6 generators 3 ISA tiers

01Why Radix-25 Matters

R=25 = 5² is the highest non-power-of-2 workhorse radix. It eliminates two factor-of-5 stages in a single codelet call. The planner uses it aggressively:

NFactorizationStagesR=25 role
20025 × 82Outer (K=8)
100025 × 10 × 43Outer (K=40)
500025 × 25 × 83Two stages (K=200, K=8)
1000025 × 25 × 163Two stages (K=400, K=16)
2000025 × 25 × 323Two stages (K=800, K=32)
The N=5000 problem: Factorized as 25×25×8, the outer stage is R=25 K=200. The twiddle table is 24×200×16 = 76KB — larger than L1. This is the first size where the planner's block-walk engine activates. If the walk engine or dispatch has a bug at K=200, N=5000 is the canary.

Unlike R=32 and R=64 which primarily serve power-of-2 transforms, R=25 appears whenever N has a factor of 25 (or 5², or any even power of 5). Combined with R=10 and R=5, these three radixes cover all multiples of 5. R=25 is preferred over two R=5 stages because a single fused 5×5 codelet has fewer total FP operations (shared sub-expressions) and half the memory passes.

02The Two Engines

5×5 Cooley-Tukey

Decompose DFT-25 as 5 radix-5 DFTs on rows, 20 internal W25 twiddle multiplications, then 5 radix-5 DFTs on columns.

External twiddles via log₃: load W¹ and W⁵, derive W², W³, W⁴ (columns) and W¹⁰, W¹⁵, W²⁰ (rows) — 2 table loads + 6 cmuls.

Production codelet at all K values. Flat strided access, no ladder needed (24 elements fit L1 up to K≈170).

genfft DAG

Translated from FFTW's t1_25.c via translate_genfft_tw.py. FMA variant: 84 add + 48 mul + 316 FMA = 448 ops. 138 stack variables, 47 constants.

Benchmarks show it ties 5×5CT at K≤32 and loses at K≥64. Included as scalar fallback for odd K values (K=25 from N=5000=25²×8).

Constants prefixed R25D_ to avoid shadowing mono IL's KP locals.

04Log₃ Twiddle Derivation for R=25

External twiddles Wn for n=1..24 are organized as a 5×5 grid: n = 5·row + col. Only W¹ and W⁵ are loaded from the table per k-step. Six bases are derived via three levels of squaring and multiplication: 

/* Load from table: 2 strided loads per k-step */
ew1 = load(tw_re[0*K+k])    // W¹
ew5 = load(tw_re[4*K+k])    // W⁵

/* Column bases: W² .. W⁴ (3 cmuls) */
ew2 = ew1 × ew1              // W² = (W¹)²
ew3 = ew1 × ew2              // W³
ew4 = ew2 × ew2              // W⁴ = (W²)²

/* Row bases: W¹⁰, W¹⁵, W²⁰ (3 cmuls) */
ew10 = ew5  × ew5            // W¹⁰ = (W⁵)²
ew15 = ew5  × ew10           // W¹⁵
ew20 = ew10 × ew10           // W²⁰ = (W¹⁰)²

/* Apply: W^n = W^col × W^(5·row) — 2 cmuls for general n */
/* Special: n%5==0 → 1 cmul, n<5 → 1 cmul, n==0 → free */

Cost: 2 table loads + 6 cmuls (derivation) + up to 2 cmuls per element (application). For the 20 non-trivial twiddle elements: ~40 cmuls total.

Benefit: 2 strided memory accesses instead of 48. At K=200 this means 2×1.6KB stride instead of 48×1.6KB — the L1 prefetcher handles 2 streams trivially.

Compared to R=64's log₃: R=25 has 8 bases (vs 14), 6 derivation cmuls (vs 12), and 24 twiddle elements (vs 63). The grid is 5×5 (vs 8×8). Same principle, simpler math. No lean/full split needed — 8 bases = 16 YMM, fitting AVX2's budget comfortably alongside the radix-5 butterfly.

055×5 Butterfly Structure

DFT-25 decomposes as a 5×5 matrix: 5 row-DFT-5s, internal W25 twiddles, then 5 column-DFT-5s. Each DFT-5 uses 4 complex multiplications and 10 additions. The internal twiddles W25n·m for n=1..4, m=1..4 are 16 non-trivial constants plus 4 ×j rotations handled implicitly.

PropertyValue

Unlike R=64 which needs a split butterfly on AVX2 (peak 33 YMM on 16 regs), R=25's 5×5 structure is compact. The radix-5 sub-butterfly uses 10 data regs (5 re + 5 im), and with 8 log₃ bases + 3 constants = ~12 peak. This fits naturally on both AVX2 (16 YMM) and AVX-512 (32 ZMM) without any ISA-specific butterfly tricks.

06Block-Walk at K=200

N=5000 = 25×25×8 produces an outer stage with R=25, K=200. The twiddle table is 24×200×16 = 76KB — well past the 32KB walk threshold. The planner activates the block-walk engine.

Walk State for R=25

typedef struct {
    size_t R, T, K;
    double *walk_re, *walk_im;    // 24 × T = 96 doubles (AVX2: T=4)
    double *step_re, *step_im;    // 24 doubles each
} vfft_walk_state;                // ~960 bytes total

Per-Block Pipeline at K=200, T=4

nb = 200 / 4 = 50 blocks

for (b = 0; b < 50; b++) {
    // 1. Pack: 25 strided loads → contiguous 25×4 = 100 doubles
    for (n = 0; n < 25; n++)
        copy 4 doubles from in[n*200 + b*4] → block[n*4];

    // 2. Kernel: flat 5×5CT at K=4 (all data in L1)
    radix25_tw_flat_dit_kernel_fwd_avx2(block, walk_tw, K=4);

    // 3. Unpack: contiguous → strided output

    // 4. Walk: rotate 24 accumulators by step angle
    for (n = 0; n < 24; n++)
        acc[n] = cmul(acc[n], step[n]);  // 24 scalar cmuls
}

Walk boundary: R=25's walk threshold is at K ≈ 86 (24×86×16 = 33KB). Sizes below this (N=200, N=400, N=1000, N=2000) use flat access. Sizes at or above (N=5000 with K=200) use walk. This boundary is the first activation of walk in the test suite and must be verified carefully.

11Design Decisions

Why not wire the DAG into dispatch?

Benchmarks showed 5×5CT ties or beats the DAG at every K. R=25's 5×5 structure is compact enough (peak ~12 YMM) that the structured register reuse of the CT codelet has no overhead disadvantage vs the DAG's fused schedule. Same conclusion as R=32. The DAG is included as a reference implementation and as the scalar fallback for the translate_genfft_tw.py generator.

Why does R=25 need scalar fallback?

N=5000 = 25×25×8 produces stages with K=200 and K=8. The K=200 stage walks, so the codelet sees K=4 (or K=8 on AVX-512) — SIMD-aligned. But N=625 = 25×25 produces K=25 at the outer stage. K=25 is not a multiple of 4 — neither AVX2 nor AVX-512 can handle it. The scalar fallback ensures correctness for any K the planner produces.

Why is R=25 preferred over two R=5 stages?

A single R=25 codelet does one pass over R×K data with fused twiddles. Two R=5 stages do two passes (with an intermediate buffer swap) and apply separate twiddle tables. The fused R=25 pass saves: one full-array memcpy, one twiddle table allocation, and shares sub-expressions within the 5×5 butterfly that two separate R=5 butterflies cannot exploit. The planner greedily extracts R=25 factors before R=5.

Why log₃ instead of direct load?

R=25 has 24 twiddle elements. Direct loading means 48 strided accesses per k-step (24 re + 24 im). Log₃ reduces this to 2 loads + 6 cmuls. At K=200 (stride 1.6KB), the 46 saved cache misses far outweigh 6 multiply instructions. Even at small K where L1 holds the table, log₃ only costs 12 FMA instructions — negligible vs the butterfly's 380 ops.

Where does walk activate?

Walk threshold: tw_table > 32KB → K > 32768/(24×16) ≈ 86. In the test suite, the first activation is N=5000 with K=200. Sizes N=200 (K=8), N=400 (K=16), N=1000 (K=40), N=2000 (K=80) all stay below the threshold and use flat access. N=5000 is the boundary size where correctness of the walk engine matters.
